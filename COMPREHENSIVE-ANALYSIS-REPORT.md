## 📊 概念定义与属性关系

### 核心概念定义表

| 概念 | 定义 | 属性 | 关系 |
|------|------|------|------|
| **Entity** | 有唯一标识的对象 | ID、连续性、可变性 | 属于 Aggregate，有领域事件 |
| **Value Object** | 无标识，属性决定相等性 | 不可变、可替换 | 属于 Entity 或 Aggregate |
| **Aggregate** | 一致性边界内的实体组 | 根实体、边界、事务一致性 | 包含 Entity 和 VO |
| **Repository** | 聚合的持久化抽象 | 集合语义、持久化透明 | 依赖 Aggregate |
| **Domain Event** | 领域中发生的业务事件 | 时态性、不可变、可订阅 | 由 Entity 产生 |
| **Specification** | 业务规则的谓词封装 | 可组合、可复用、可测试 | 应用于 Entity |

## 🔍 论证与反例分析

### 论证 1: Clean Architecture 能够降低复杂度

**前提**:

- P1: 不同变化速率的代码应该分离
- P2: 业务规则变化慢，框架变化快
- P3: 分离关注点可以降低复杂度

**推理**:

```
P1 ∧ P2 ──► 分层隔离满足不同变化速率
P3 ∧ (分层隔离) ──► 降低复杂度
───────────────────────────────────
∴ Clean Architecture 降低复杂度
```

**反例检验**:

- 场景: 简单 CRUD 应用
- 反例: 使用 Clean Architecture 会增加不必要的抽象
- 结论: 简单场景不适用，但复杂业务系统论证成立

### 论证 2: eBPF 实现低开销监控

**前提**:

- P1: 上下文切换有性能开销
- P2: eBPF 在内核态执行，无需切换到用户态
- P3: 性能开销与上下文切换次数成正比

**推理**:

```
P1 ∧ P2 ──► 节省上下文切换开销
P3 ∧ (节省切换) ──► 低开销
───────────────────────────────────
∴ eBPF 监控低开销
```

**数据支持**:

- 传统观测: ~5-10% CPU 开销
- eBPF 观测: ~0.5-1% CPU 开销
- 来源: CNCF eBPF 报告 2024

### 论证 3: ABAC 比 RBAC 更适合复杂授权

**前提**:

- P1: RBAC 授权粒度 = f(Role)，粗粒度
- P2: ABAC 授权粒度 = f(S,R,A,E)，细粒度
- P3: 复杂场景需要细粒度授权

**推理**:

```
Granularity(ABAC) > Granularity(RBAC)
ComplexScenes ──► Need(FineGrained)
───────────────────────────────────
∴ ABAC 更适合复杂授权场景
```

**反例检验**:

- 场景: 简单内部系统，角色固定
- 反例: ABAC 增加复杂度，RBAC 足够
- 结论: 简单场景 RBAC 更合适，复杂场景 ABAC 论证成立
