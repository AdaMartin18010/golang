# Go代码审查指南

> 建立有效的代码审查流程与检查清单

---

## 一、代码审查的价值

### 1.1 为什么需要代码审查

```text
代码审查的多重价值：
────────────────────────────────────────

1. 缺陷预防：
   - 在合并前发现bug
   - 成本远低于生产环境修复
   - 研究表明能发现60-90%的缺陷

2. 知识共享：
   - 团队成员互相学习
   - 了解项目不同部分
   - 减少"只有某人懂"的风险

3. 代码一致性：
   - 统一编码风格
   - 遵循设计约定
   - 维护代码质量基线

4. 安全审查：
   - 发现安全漏洞
   - 检查敏感数据处理
   - 验证权限控制

成本效益分析：
────────────────────────────────────────

修复成本随阶段增长：

┌─────────────┬────────────────┐
│   阶段      │    相对成本     │
├─────────────┼────────────────┤
│ 开发时      │     1x         │
│ 代码审查    │    10x         │
│ 测试时      │   100x         │
│ 生产环境    │  1000x+        │
└─────────────┴────────────────┘

代码审查投入1小时，
可能节省测试和生产环境数小时的修复时间。

审查效果数据：
────────────────────────────────────────

研究统计：
- 平均每次审查发现约3-5个问题
- 70%是风格/最佳实践问题
- 20%是潜在bug
- 10%是安全问题

审查密度：
- 每小时审查200-400行代码效果最佳
- 超过500行，发现率显著下降
- 每次审查控制在1小时内
```

### 1.2 审查心态

```text
建设性反馈：
────────────────────────────────────────

不良的审查评论：
❌ "这段代码太烂了"
❌ "谁教你这么写的？"
❌ "明显有bug"

建设性的评论：
✅ "这里可能有并发问题，建议加锁"
✅ "建议提取这个逻辑到单独的函数"
✅ "有没有考虑使用标准库的函数？"

提问而非指责：
────────────────────────────────────────

❌ "为什么没有错误处理？"
✅ "这里如果发生错误会怎样？"

❌ "这个变量名不好"
✅ "能否用更描述性的名字？"

解释"为什么"：
────────────────────────────────────────

❌ "改用map"
✅ "改用map可以将时间复杂度从O(n)降到O(1)，
     在数据量大时性能更好"

承认不确定性：
────────────────────────────────────────

"我不确定这里是否有性能问题，
 建议运行一下基准测试看看"

"这个方法名让我困惑，
 可能是我理解错了意图"
```

## 二、审查检查清单

### 2.1 功能性检查

```text
正确性验证：
────────────────────────────────────────

□ 逻辑正确性：
  - 条件判断是否正确
  - 循环终止条件是否正确
  - 边界条件是否处理

□ 错误处理：
  - 所有错误都被检查
  - 错误被适当包装
  - 没有忽略的错误

□ 并发安全：
  - 共享数据有适当的锁
  - 避免数据竞争
  - 无死锁风险

□ 资源管理：
  - 文件句柄关闭
  - 数据库连接释放
  - goroutine不会泄漏

□ 边界情况：
  - 空输入处理
  - 最大值/最小值
  - 异常情况

测试检查：
────────────────────────────────────────

□ 测试覆盖：
  - 新增代码有测试
  - 边界情况有测试
  - 错误路径有测试

□ 测试质量：
  - 测试独立，不依赖顺序
  - 测试描述清晰
  - 使用表格驱动测试
```

### 2.2 代码质量检查

```text
可读性：
────────────────────────────────────────

□ 命名清晰：
  - 函数名描述行为
  - 变量名有意义
  - 避免缩写

□ 函数长度：
  - 单一职责
  - 长度适中（<50行）
  - 参数数量合理（<5个）

□ 注释质量：
  - 解释"为什么"而非"做了什么"
  - 复杂逻辑有注释
  - 公开API有文档

可维护性：
────────────────────────────────────────

□ 代码组织：
  - 相关代码在一起
  - 依赖关系清晰
  - 避免循环依赖

□ 重复消除：
  - 无重复代码
  - 使用通用抽象
  - 提取公共逻辑

□ 复杂度控制：
  - 嵌套层级不深
  - 避免过多if/else
  - 简化复杂表达式

性能考量：
────────────────────────────────────────

□ 算法效率：
  - 时间复杂度合理
  - 避免不必要的循环
  - 使用合适的数据结构

□ 内存使用：
  - 避免内存泄漏
  - 减少不必要的分配
  - 预分配slice容量
```

### 2.3 Go特定检查

```text
语言习惯：
────────────────────────────────────────

□ 错误处理：
  ```go
  // 正确
  if err != nil {
      return err
  }

  // 避免
  if err == nil {
      // ...
  }
  ```

□ 空值检查：

  ```go
  // 正确
  if s == "" {
  }
  if len(slice) == 0 {
  }

  // 避免
  if s != "" {
      // ...
  } else {
      return errors.New("empty")
  }
  ```

□ defer使用：

  ```go
  // 正确
  f, err := os.Open(file)
  if err != nil {
      return err
  }
  defer f.Close()

  // 避免在循环中使用defer
  for _, file := range files {
      f, _ := os.Open(file)
      defer f.Close()  // 循环结束后才执行！
  }
  ```

□ 并发模式：

  ```go
  // 正确：使用Context控制
  ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
  defer cancel()

  // 避免：裸goroutine
  go func() {
      // 无法控制、无法取消
  }()
  ```

标准库使用：
────────────────────────────────────────

□ 优先使用标准库：

  ```go
  // 正确
  strings.HasPrefix(s, prefix)

  // 避免
  s[:len(prefix)] == prefix
  ```

□ 使用path/filepath而非path：

  ```go
  // Windows兼容
  filepath.Join("dir", "file.txt")
  ```

□ 使用time.Duration：

  ```go
  // 正确
  timeout := 5 * time.Second

  // 避免
  timeout := 5000  // 毫秒？微秒？
  ```

```

## 三、审查流程

### 3.1 提交前自检

```

作者检查清单：
────────────────────────────────────────

提交前自问：

□ 我是否测试了这些更改？

- 单元测试通过
- 集成测试通过
- 手动验证功能

□ 代码是否干净？

- 无调试代码
- 无注释掉的代码
- 无TODO（除非有对应issue）

□ 提交信息是否清晰？

- 描述做了什么
- 解释为什么做
- 关联相关issue

□ 是否有文档更新？

- API文档
- README更新
- 配置变更说明

最小化提交：
────────────────────────────────────────

良好实践：

- 每个提交只做一件事
- 提交可以独立审查
- 便于回滚和cherry-pick

避免：

- 一个提交包含多个功能
- 代码格式化与逻辑修改混在一起
- 提交包含无关文件

```

### 3.2 审查流程

```text

标准审查步骤：
────────────────────────────────────────

1. 理解背景：
   - 阅读PR描述
   - 了解变更原因
   - 查看相关issue

2. 整体浏览：
   - 文件变更列表
   - 理解架构变化
   - 检查测试覆盖

3. 详细审查：
   - 逐行阅读代码
   - 检查逻辑正确性
   - 验证边界处理

4. 运行验证：
   - 本地运行测试
   - 检查lint通过
   - 验证功能（如需要）

5. 提供反馈：
   - 分类问题严重性
   - 提供改进建议
   - 认可好的实践

反馈分类：
────────────────────────────────────────

🔴 必须修复（Blocking）：

- 安全漏洞
- 明显bug
- 性能严重问题
- 测试缺失

🟡 建议修改（Non-blocking）：

- 代码风格问题
- 小优化建议
- 命名改进

🟢 正面反馈（Positive）：

- 好的设计
- clever的解决方案
- 完善的测试

审查回应：
────────────────────────────────────────

收到反馈后：

1. 不要防御性回应
2. 询问不理解的建议
3. 解释设计决策
4. 及时修复问题
5. 标记已修复的评论

示例回应：
"感谢指出，已修复"
"好的，我改成使用标准库函数"
"这里我保持原样，因为..."

```
