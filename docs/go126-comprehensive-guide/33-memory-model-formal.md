# Go内存模型形式化规范

> 基于Happens-before关系的Go内存模型数学规范

---

## 一、Happens-before关系公理系统

### 1.1 基本公理

```text
定义 1.1 (事件):
────────────────────────────────────────
事件 e ∈ Event 表示程序执行中的原子操作

事件类型：
- Write(x, v)：写入变量x值为v
- Read(x, v)：从变量x读取值v
- Lock(m)：获取互斥锁m
- Unlock(m)：释放互斥锁m
- Send(ch, v)：向channel ch发送值v
- Recv(ch, v)：从channel ch接收值v
- Close(ch)：关闭channel ch
- Fork(g)：创建goroutine g
- Join(g)：等待goroutine g结束

定义 1.2 (Happens-before):
────────────────────────────────────────
二元关系 ≺ ⊆ Event × Event

表示事件e₁发生在事件e₂之前，且e₁对e₂可见

公理 1.1 (程序顺序):
────────────────────────────────────────
∀e₁, e₂ ∈ Event,
若e₁和e₂在同一线程(或goroutine)中，
且e₁在程序顺序上先于e₂，
则 e₁ ≺ e₂

形式化：
SameThread(e₁, e₂) ∧ ProgramOrder(e₁, e₂) ⟹ e₁ ≺ e₂

公理 1.2 (传递性):
────────────────────────────────────────
∀e₁, e₂, e₃ ∈ Event,
e₁ ≺ e₂ ∧ e₂ ≺ e₃ ⟹ e₁ ≺ e₃

公理 1.3 (非自反性):
────────────────────────────────────────
∀e ∈ Event, ¬(e ≺ e)

由传递性和非自反性，≺是严格偏序关系

公理 1.4 (锁顺序):
────────────────────────────────────────
∀m ∈ Mutex, ∀e₁, e₂ ∈ Event,
e₁ = Unlock(m) ∧ e₂ = Lock(m) ∧
SameMutex(e₁, e₂) ∧ Time(e₁) < Time(e₂)
⟹ e₁ ≺ e₂

即：对同一mutex的解锁发生在后续加锁之前
```

### 1.2 Channel操作公理

```text
公理 1.5 (无缓冲Channel同步):
────────────────────────────────────────
∀ch ∈ Channel, ∀e₁, e₂ ∈ Event,
(e₁ = Send(ch, v) ∧ e₂ = Recv(ch, v) ∧ Matched(e₁, e₂))
⟹ e₁ ≺ e₂

即：无缓冲channel的发送happens-before匹配的接收

证明：
无缓冲channel的发送方会阻塞直到接收方就绪，
发送操作在接收操作完成前完成，
故发送 ≺ 接收。∎

公理 1.6 (有缓冲Channel同步):
────────────────────────────────────────
∀ch ∈ Channel, cap(ch) = n > 0,
∀i ∈ ℕ,
e₁ = Sendᵢ(ch, v) ∧ e₂ = Recvᵢ(ch, v)
⟹ e₁ ≺ e₂

即：第i个发送happens-before第i个接收
(当缓冲满时)

公理 1.7 (Channel关闭):
────────────────────────────────────────
∀ch ∈ Channel,
e₁ = Close(ch) ∧ e₂ = Recv(ch, zero) ∧ returnsClosed(e₂)
⟹ e₁ ≺ e₂

即：关闭channel happens-before任何检测到的关闭

定理 1.1 (Channel传递性):
────────────────────────────────────────
若 e₁ ≺ Send(ch) ∧ Recv(ch) ≺ e₂
则 e₁ ≺ e₂

证明：
由公理1.5，Send(ch) ≺ Recv(ch)
由传递性，e₁ ≺ Send(ch) ≺ Recv(ch) ≺ e₂
故 e₁ ≺ e₂ ∎
```

### 1.3 原子操作公理

```text
公理 1.8 (原子写读):
────────────────────────────────────────
∀x ∈ Variable,
e₁ = AtomicWrite(x, v) ∧ e₂ = AtomicRead(x, v)
⟹ e₁ ≺ e₂

即：原子写happens-before读取相同值的原子读

注意：这不保证非原子读，也不保证读不同值的情况

公理 1.9 (顺序一致性):
────────────────────────────────────────
所有原子操作构成一个全局的全序关系
即：存在一个全序 < 包含所有原子操作，
且该全序与每个goroutine的程序顺序一致

形式化：
∃ < : TotalOrder,
∀e₁, e₂ ∈ AtomicEvents,
SameThread(e₁, e₂) ∧ ProgramOrder(e₁, e₂) ⟹ e₁ < e₂

定理 1.2 (原子操作可见性):
────────────────────────────────────────
若 e₁ = AtomicWrite(x, v) 且 e₁ < e₂
则 e₂ 读x时看到v或更晚的值

即：原子操作的全序保证了全局可见性
```

## 二、数据竞争的形式化定义

### 2.1 竞争条件定义

```text
定义 2.1 (数据竞争):
────────────────────────────────────────
两个内存访问e₁和e₂构成数据竞争，当且仅当：

1. e₁和e₂访问同一内存位置
2. 至少一个是写操作
3. e₁和e₂在不同goroutine中
4. ¬(e₁ ≺ e₂) ∧ ¬(e₂ ≺ e₁)
   (二者无happens-before关系)

形式化：
Race(e₁, e₂) ⟺
  SameLocation(e₁, e₂) ∧
  (IsWrite(e₁) ∨ IsWrite(e₂)) ∧
  DifferentGoroutine(e₁, e₂) ∧
  ¬(e₁ ≺ e₂) ∧ ¬(e₂ ≺ e₁)

定理 2.1 (数据竞争检测完备性):
────────────────────────────────────────
Go的race detector基于happens-before关系，
能够检测所有实际发生的数据竞争。

证明思路：
- race detector维护向量时钟追踪happens-before
- 对于任何两个无hb关系的内存访问，都能检测到
- 虽然存在误报可能(未实际执行的路径)，但无漏报∎

定义 2.2 (同步关系):
────────────────────────────────────────
同步关系 Sync ⊆ Event × Event

Sync包含所有happens-before关系的直接来源：
1. 程序顺序同步
2. 锁操作同步
3. Channel操作同步
4. 原子操作同步
5. Once操作同步

定理 2.2 (同步传递性):
────────────────────────────────────────
Happens-before关系是同步关系的传递闭包：
≺ = Sync⁺

即：e₁ ≺ e₂ 当且仅当存在同步链连接e₁和e₂
```

### 2.2 避免数据竞争的模式

```text
定理 2.3 (Mutex保护):
────────────────────────────────────────
若所有对变量x的访问都持有同一mutex m，
则对x的访问无数据竞争。

证明：
设e₁和e₂是任意两个对x的访问
由于都持有m，设它们对应的锁操作为：
Lock₁(m), Unlock₁(m) 包裹 e₁
Lock₂(m), Unlock₂(m) 包裹 e₂

由锁语义，操作序列必为：
... Unlockᵢ(m) ... Lockⱼ(m) ... (i ≠ j)

由公理1.4，Unlockᵢ(m) ≺ Lockⱼ(m)
故 e₁ ≺ Unlockᵢ(m) ≺ Lockⱼ(m) ≺ e₂
或 e₂ ≺ e₁

因此e₁和e₂有hb关系，不构成竞争∎

定理 2.4 (Channel通信):
────────────────────────────────────────
若对变量x的所有写操作都通过Send发送到channel ch，
且对x的所有读操作都从ch接收，
则对x的访问无数据竞争。

证明：
设写操作e₁通过Send₁(ch)发送
读操作e₂通过Recv₂(ch)接收

由公理1.5，Send₁(ch) ≺ Recv₂(ch)
若写发生在读之前，则e₁ ≺ Send₁ ≺ Recv₂ ≺ e₂
故无竞争∎

定理 2.5 (原子变量):
────────────────────────────────────────
若对变量x的所有访问都使用原子操作，
则虽然可能存在竞争，但行为是良定义的
(由顺序一致性保证)
```

## 三、内存序模型

### 3.1 宽松内存序

```text
定义 3.1 (内存序):
────────────────────────────────────────
程序序 (Program Order, po)：
同一线程内语句的执行顺序

观察序 (Observation Order, obs)：
一个读操作观察到的写操作的顺序

修改序 (Modification Order, mo)：
对同一原子变量的所有修改操作的全序

 sequenced-before (sb)：
求值顺序关系，同一线程内的操作序列

 happens-before (hb)：
如前文定义的偏序关系

 synchronizes-with (sw)：
同步关系，直接的跨线程顺序

定义 3.2 (sequenced-before):
────────────────────────────────────────
sb是单个线程内求值的严格偏序关系

若表达式e₁的值计算在表达式e₂的副作用之前，
且e₁和e₂在同一线程，则 e₁ sb e₂

定理 3.1 (sb与hb关系):
────────────────────────────────────────
同一线程内：e₁ sb e₂ ⟹ e₁ hb e₂

但反之不成立：hb可以跨线程，sb只在单线程内
```

### 3.2 可见性语义

```text
定义 3.3 (可见性):
────────────────────────────────────────
写操作w对读操作r可见，当且仅当：
1. w hb r，或
2. 存在写操作w'使得w' hb r，且w' mo w
   (w'在修改序上在w之前)

直观：
- 如果写发生在读之前(hb)，则读可见该写
- 如果读看到的不是最新值，那是因为更早的写被覆盖了

定义 3.4 (读-写一致性):
────────────────────────────────────────
读操作r读取的值必须来自某个写操作w，满足：
1. w对r可见
2. 不存在另一个写w'使得w hb w' hb r
   (没有更近的写)

定理 3.2 (读值唯一性):
────────────────────────────────────────
在无数据竞争的程序中，每个读操作看到唯一的值

在有数据竞争的程序中，读操作可能看到不确定的值
```

## 四、形式化验证应用

### 4.1 程序验证

```text
霍尔三元组在并发中的扩展:
────────────────────────────────────────
{P} C {Q} 表示：
若前置条件P成立，执行命令C后，后置条件Q成立

并发霍尔逻辑增加：
- 资源断言 (resource assertions)
- 分离合取 (separating conjunction)
- 不变式 (invariants)

示例：
{ x ↦ _ }
  x := 1
{ x ↦ 1 }

表示：x初始指向某个值，赋值后x指向1

分离合取 P * Q：
P和Q持有的资源不相交

帧规则 (Frame Rule):
{P} C {Q} 且 R与C无关
─────────────────────────
{P * R} C {Q * R}
```

### 4.2 模型检测

```text
模型检测方法:
────────────────────────────────────────

将Go程序转换为状态转换系统：
S = (States, Initial, Transitions)

其中：
- States：程序状态集合
- Initial：初始状态
- Transitions：状态转换关系

验证性质：
□(NoRace) ：始终无数据竞争
◇(Termination) ：最终终止
□(DeadlockFree) ：始终无死锁

状态爆炸问题：
并发程序状态空间随goroutine数指数增长

缓解方法：
1. 偏序规约 (POR)
2. 对称性规约
3. 抽象解释
```
