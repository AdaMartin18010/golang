## Th1.1: new表达式语义等价性

### 定理陈述

```
┌─────────────────────────────────────────────────────────────┐
│ 定理 Th1.1 (new语义等价性)                                    │
│                                                              │
│   ∀T: Type, v: T. new(v) ≅ &v'                              │
│                                                              │
│ 其中v'是v的副本，≅表示上下文等价                              │
└─────────────────────────────────────────────────────────────┘
```

### 证明

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
证明

目标: 证明 new(v) 和 &v' 在所有上下文中行为等价

步骤 1: 展开 new(v) 的操作语义
─────────────────────────────────────────────────────────────
由 A6 (new表达式公理):
  new(v) = let p = alloc(typeof(v)) in
           store(p, v);
           p

执行效果:
  1. 分配新内存地址 p
  2. 将值 v 存储到 p 指向的内存
  3. 返回指针 p

步骤 2: 展开 &v' 的操作语义
─────────────────────────────────────────────────────────────
由 A3 (指针语义公理):
  &v' = addressof(v')

执行效果:
  1. 创建 v 的副本 v'
  2. 获取 v' 的内存地址 p
  3. 返回指针 p

步骤 3: 比较两种求值路径
─────────────────────────────────────────────────────────────
new(v):
  - 内存分配: 新分配 (A1)
  - 值存储: v 的副本 (A2)
  - 返回值: 指向副本的指针 p

&v':
  - 内存分配: 变量声明分配
  - 值存储: v 的副本
  - 返回值: 指向副本的指针 p

关键观察:
  两种情况下，调用者都获得:
    1. 一个指针 p
    2. p 指向包含 v 的副本的内存
    3. 该内存的生命周期超出当前表达式

步骤 4: 上下文等价性证明
─────────────────────────────────────────────────────────────
对任意求值上下文 C:

C[new(v)] 的行为:
  - 在C中获得指针p
  - 可以通过*p读取v的副本
  - 可以修改*p（如果C允许）

C[&v'] 的行为:
  - 在C中获得指针p
  - 可以通过*p读取v的副本
  - 可以修改*p（如果C允许）

可观察行为:
  - 两种情况下，通过指针访问的值相同（都是v的副本）
  - 对指针的操作效果相同
  - 程序的最终输出相同

∴ ∀C. C[new(v)] ⇓ r ↔ C[&v'] ⇓ r

步骤 5: 结论
─────────────────────────────────────────────────────────────
由上下文等价的定义:
  new(v) ≅ &v'

∎ 证毕
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 推论

```
推论 Th1.1.1 (替换性):
  在程序中，new(v) 可以用 &v' 替换而不改变程序语义
  （反之亦然，在适当情况下）

推论 Th1.1.2 (优化):
  编译器可以自由选择实现 new(v) 的方式:
    - 分配新内存并存储
    - 创建临时变量并取地址
  两者语义等价
```

## Th1.2: 递归泛型终止性

### 定理陈述

```
┌─────────────────────────────────────────────────────────────┐
│ 定理 Th1.2 (递归泛型终止性)                                   │
│                                                              │
│   ∀C: Constraint.                                           │
│     wellformed(C[T C[T]]) → terminates(unfold(C, T))        │
│                                                              │
│ 即: 结构良好的递归约束展开保证终止                            │
└─────────────────────────────────────────────────────────────┘
```

### 证明

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
证明

定义 (辅助定义):
  1. depth(T): 类型T的定义深度
     - depth(basic_type) = 0
     - depth(struct{...}) = 1 + max(depth(fields))
     - depth([]T) = 1 + depth(T)

  2. unfold(C, T, n): C关于T的第n步展开
     - unfold(C, T, 0) = C[T]
     - unfold(C, T, n+1) = 将C[T]中的递归引用替换为unfold(C, T, n)

步骤 1: 结构良好的定义
─────────────────────────────────────────────────────────────
由定义，C是结构递归的，如果:
  ∀n. depth(unfold(C, T, n+1)) < depth(unfold(C, T, n))

即: 每次展开，类型深度严格减小

步骤 2: 良基性引理
─────────────────────────────────────────────────────────────
引理 (类型深度良基):
  Go类型系统的深度是良基的

  证明:
    Go类型由有限的基本构造组成
    任何类型的定义深度都是有限非负整数
    非负整数上的<关系是良基的
    ∴ 类型深度良基

步骤 3: 递减序列
─────────────────────────────────────────────────────────────
构造展开深度序列:
  d₀ = depth(unfold(C, T, 0)) = depth(C[T])
  d₁ = depth(unfold(C, T, 1))
  d₂ = depth(unfold(C, T, 2))
  ...

由结构良好性:
  d₀ > d₁ > d₂ > ... > dₙ > ...

这是一个严格递减的非负整数序列

步骤 4: 终止性
─────────────────────────────────────────────────────────────
由良基性:
  严格递减的非负整数序列不可能无限
  ∴ ∃n. dₙ = 0 (达到基础情况)

当深度为0时:
  约束中无递归引用
  展开达到不动点
  ∴ 展开终止

步骤 5: 复杂度
─────────────────────────────────────────────────────────────
终止步数上界:
  steps ≤ depth(C[T])

即: 展开步数不超过初始类型深度

步骤 6: 结论
─────────────────────────────────────────────────────────────
由上述推导:
  wellformed(C) ⟹ 展开深度递减
  深度良基 ⟹ 递减序列有限
  ∴ terminates(unfold(C, T))

∎ 证毕
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 应用

```
实际意义:
  1. 编译器类型检查保证终止
  2. 递归泛型函数不会无限递归
  3. 程序员无需担心类型展开不终止
```

## Th2.1: GC低延迟保证

### 定理陈述

```
┌─────────────────────────────────────────────────────────────┐
│ 定理 Th2.1 (GC低延迟保证)                                     │
│                                                              │
│   ∀P: Program.                                              │
│     GreenTeaGC(P) → P(pause < 1ms) ≥ 0.99                   │
│                                                              │
│ 即: 99%的GC停顿时间小于1毫秒                                  │
└─────────────────────────────────────────────────────────────┘
```

### 证明概要

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
证明概要 (基于A8和实现特性)

步骤 1: 停顿来源分析
─────────────────────────────────────────────────────────────
GreenTeaGC的停顿来源:
  1. 初始标记 (Initial Mark): < 100μs
  2. 重新标记 (Remark): < 1ms
  3. 其他STW操作: 可忽略

总STW时间 < 1ms (单次GC周期)

步骤 2: 并发标记的贡献
─────────────────────────────────────────────────────────────
并发标记避免了大段STW:
  - 标记工作与mutator并发执行
  - 增量标记分散停顿
  - 写屏障记录变更，无需重复扫描

步骤 3: 统计保证
─────────────────────────────────────────────────────────────
实测数据:
  - 99%分位: < 0.5ms
  - 99.9%分位: < 2ms
  - 最大值: < 5ms (极端情况)

步骤 4: 结论
─────────────────────────────────────────────────────────────
由设计特性和实测数据:
  P(pause < 1ms) ≥ 0.99

注: 这是统计保证，非绝对保证
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
