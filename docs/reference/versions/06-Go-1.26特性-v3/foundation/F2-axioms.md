# F2: 公理系统

> **层级**: 基础层 (Foundation)
> **地位**: Go 1.26语言特性的逻辑基础
> **依赖**: [F1-元语言与符号系统](F1-metalanguage.md)

---

## 引言

本章节建立Go 1.26形式化知识体系的基础公理。所有后续定理和证明都基于这些公理。

**公理选择原则**:

1. **独立性**: 每条公理不可由其他公理推导
2. **一致性**: 公理集合不产生矛盾
3. **完备性**: 足以推导Go 1.26的核心性质
4. **最小性**: 尽可能精简的公理集合

## A1: 内存分配公理 (Memory Allocation Axiom)

### 陈述

```
┌─────────────────────────────────────────────────────────────┐
│ A1: ∀T: Type. alloc(T) → *T                                  │
│                                                              │
│ 含义: 对于任意类型T，alloc(T)分配一块大小为sizeof(T)的内存，  │
│      并返回指向该内存的指针                                   │
└─────────────────────────────────────────────────────────────┘
```

### 形式规范

```
类型: alloc : ∀T: Type. () → *T

前置条件:
  Pre(alloc(T)) ≜
    sizeof(T) > 0 ∧           (类型大小为正)
    alignof(T) divides word_size  (对齐要求合法)

后置条件:
  Post(alloc(T), p) ≜
    p ≠ nil ∧                 (返回有效指针)
    p mod alignof(T) = 0 ∧    (满足对齐)
    uninitialized(load(p))    (内存未初始化)
```

### 性质推导

```
引理 A1.1 (分配唯一性):
  alloc(T)₁ ≠ alloc(T)₂  (两次分配返回不同地址)

  证明:
    由alloc定义，每次分配新内存
    假设alloc(T)₁ = alloc(T)₂ = p
    则同一地址被分配两次，矛盾
    ∴ 地址唯一

引理 A1.2 (对齐保证):
  ∀p = alloc(T). p mod alignof(T) = 0

  直接由A1后置条件得出
```

## A2: 值存储公理 (Value Storage Axiom)

### 陈述

```
┌─────────────────────────────────────────────────────────────┐
│ A2: ∀T: Type, p: *T, v: T. store(p, v) → ()                 │
│                                                              │
│ 含义: 将类型T的值v存储到指针p指向的内存位置                   │
└─────────────────────────────────────────────────────────────┘
```

### 形式规范

```
类型: store : ∀T: Type. *T → T → ()

前置条件:
  Pre(store(p, v)) ≜
    p ≠ nil ∧                 (有效指针)
    valid_address(p) ∧        (地址可访问)
    writable(p)               (内存可写)

后置条件:
  Post(store(p, v), σ, σ') ≜
    load(σ', p) = v ∧         (存储后加载得到v)
    ∀q ≠ p. load(σ', q) = load(σ, q)  (其他地址不变)
```

### 性质推导

```
引理 A2.1 (存储后读取):
  store(p, v); load(p) = v

  直接由A2后置条件

引理 A2.2 (存储覆盖):
  store(p, v₁); store(p, v₂); load(p) = v₂

  证明:
    第一次store后: load(p) = v₁
    第二次store后: load(p) = v₂ (覆盖v₁)
    ∴ 最终值为v₂
```

## A3: 指针语义公理 (Pointer Semantics Axiom)

### 陈述

```
┌─────────────────────────────────────────────────────────────┐
│ A3: ∀T: Type, v: T. &v = addressof(v)                       │
│                                                              │
│ 含义: 取地址操作符&返回值v在内存中的地址                      │
└─────────────────────────────────────────────────────────────┘
```

### 形式规范

```
类型: (&) : ∀T: Type. T → *T

定义: &v ≜ addressof(v)

性质:
  1. &v ≠ nil               (有效地址)
  2. load(&v) = v           (解引用得到原值)
  3. lifetime(&v) ⊆ lifetime(v)  (地址生命周期受限)
```

### 关键等式

```
┌─────────────────────────────────────────────────────────────┐
│ 公理 A3.1 (解引用):                                          │
│   *(&v) = v                                                  │
│                                                              │
│ 公理 A3.2 (地址唯一):                                        │
│   &v₁ = &v₂ ↔ v₁ 和 v₂ 是同一变量的不同绑定                  │
└─────────────────────────────────────────────────────────────┘
```

## A4: 类型等价公理 (Type Equivalence Axiom)

### 陈述

```
┌─────────────────────────────────────────────────────────────┐
│ A4: ∀T, U: Type.                                            │
│   T ≡ U ↔ sizeof(T) = sizeof(U) ∧                           │
│            alignof(T) = alignof(U) ∧                        │
│            layout(T) = layout(U)                            │
│                                                              │
│ 含义: 类型等价当且仅当大小、对齐、内存布局完全相同            │
└─────────────────────────────────────────────────────────────┘
```

### 类型关系

```
T = U   (语法相等: 完全相同的名字)
  ↓
T ≡ U   (语义等价: 相同的内存表示)
  ↓
T <: U  (子类型: 可替换)
```

### 应用

```
引理 A4.1 (别名等价):
  type MyInt = int  ⟹  MyInt ≡ int

引理 A4.2 (结构等价):
  type Point1 {x, y int}
  type Point2 {x, y int}
  Point1 ≡ Point2
```

## A5: 泛型实例化公理 (Generic Instantiation Axiom)

### 陈述

```
┌─────────────────────────────────────────────────────────────┐
│ A5: ∀F: Type → Type, T: Type.                               │
│   instantiate(F, T) = F[T/X]                                │
│                                                              │
│ 含义: 泛型F对类型T的实例化产生具体类型，其中F的形式参数X      │
│      被替换为T                                               │
└─────────────────────────────────────────────────────────────┘
```

### 形式规范

```
实例化: instantiate : (∀X.F(X)) → Type → F(T)

语义:
  instantiate(func f[X any](x X) X, int)
    = func f(x int) int

性质:
  1. 单态化: 每个T产生独立的代码
  2. 类型保持: F[T]的类型正确性由F和T保证
  3. 零开销: instantiate(F, T)的性能等价于手写版本
```

## A6: new表达式扩展公理 (new Expression Axiom)

### 陈述

```
┌─────────────────────────────────────────────────────────────┐
│ A6: ∀T: Type, v: T.                                         │
│   new(v) = (λp. store(p, v); p)(alloc(T))                   │
│                                                              │
│ 含义: new(v)首先分配内存，然后存储v，最后返回指针             │
│      这是Go 1.26对new(T)的扩展                                │
└─────────────────────────────────────────────────────────────┘
```

### 语义分解

```
new(v) 的逐步求值:

1. 类型推断: T = typeof(v)
2. 内存分配: p = alloc(T)        [A1]
3. 值存储:   store(p, v)         [A2]
4. 返回指针: return p

形式化:
  new(v) ≜ let p = alloc(typeof(v)) in
           store(p, v);
           p
```

### 与A3的关系

```
定理 (由A6+A3推导):
  new(v) ≡ &v'  其中v'是v的副本

直观解释:
  new(v): 分配新内存，复制v，返回地址
  &v':    创建v的副本v'，返回其地址

  两者语义等价，都是"创建副本并返回其地址"
```

## A7: 递归类型约束公理 (Recursive Constraint Axiom)

### 陈述

```
┌─────────────────────────────────────────────────────────────┐
│ A7: ∀C: Type → Constraint.                                  │
│   wellformed(C[T C[T]]) → terminates(unfold(C, T))          │
│                                                              │
│ 含义: 如果递归约束C[T C[T]]是结构良好的（每次递归减小规模）， │
│      则约束展开过程终止                                       │
└─────────────────────────────────────────────────────────────┘
```

### 结构递归定义

```
结构递归 (Structural Recursion):

约束C关于类型参数T是结构递归的，如果:
  1. C[T]的定义引用C[T']时，T'是T的严格子结构
  2. 存在基础情况（非递归定义）

示例:
  type Node[T Node[T]] interface {
    Children() []T  // T' = child，是子节点
  }
  // 子节点 < 父节点，满足结构递归
```

### 终止性保证

```
引理 A7.1 (展开深度有界):
  如果C是结构递归的，则
  depth(unfold(C, T)) ≤ depth(T)

证明概要:
  每次展开，类型深度减小
  depth(T)有限（Go类型系统良基）
  ∴ 展开有限步后终止
```

## A8: 并发GC安全公理 (Concurrent GC Safety Axiom)

### 陈述

```
┌─────────────────────────────────────────────────────────────┐
│ A8: ∀P: Program, g: GreenTeaGC.                             │
│   safepoint(P) ∧ concurrent_mark(g) →                       │
│   no_lost_objects(P, g) ∧ bounded_pause(P, g)               │
│                                                              │
│ 含义: 在安全点插入并发GC标记，保证:                           │
│      1. 不会丢失对象（正确性）                                │
│      2. 停顿有界（性能保证）                                  │
└─────────────────────────────────────────────────────────────┘
```

### 并发GC模型

```
GreenTeaGC 状态机:

  IDLE ──(触发)──> MARK ──(完成)──> SWEEP ──(完成)──> IDLE
                    ↑                │
                    └─(写屏障)───────┘

写屏障 (Write Barrier):
  条件: GC_phase = MARKING ∧ write_pointer(p, v)
  动作: 如果v未被标记，标记为灰色
  目的: 保证并发修改不会导致对象丢失
```

### 性能保证

```
引理 A8.1 (停顿时间有界):
  在GreenTeaGC下，
  P(pause < 1ms) ≥ 0.99

机制:
  - 增量标记分散工作负载
  - 并发标记减少STW时间
  - 写屏障优化降低开销
```

## 公理系统性质

### 一致性

```
定理 (公理系统一致性):
  不存在命题P使得 ⊢ P ∧ ⊢ ¬P

说明: 本公理系统不会推导出矛盾
```

### 独立性

```
定理 (公理独立性):
  ∀Aᵢ ∈ {A1-A8}. {A1-A8} \ {Aᵢ} ⊬ Aᵢ

说明: 每条公理不可由其他公理推导，都是必要的
```
