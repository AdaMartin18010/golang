## 📌 通用问题

### Q1: 如何升级到 Go 1.26？

**A**:

```bash
# 1. 下载安装 Go 1.26
# https://go.dev/dl/

# 2. 验证版本

# 3. 更新项目
cd your-project
go mod edit -go=1.26

# 4. 运行测试
go test ./...
```

### Q2: Go 1.26 有哪些必学的新特性？

**A**: 按优先级排序：

1. **new(expr)** - 语法糖，立即可用，提高代码可读性
2. **Green Tea GC** - 自动启用，无需学习，享受性能提升
3. **go fix** - 代码现代化工具，建议定期使用
4. **递归泛型** - 高级特性，数据结构库开发者需要掌握
5. **crypto/hpke** - 安全领域开发者需要了解

## 📝 new() 表达式

### Q3: new(expr) 和 &T{} 有什么区别？

**A**: 语义完全等价，选择取决于可读性：

```go
// 语义等价
new(T{v}) == &T{v}
new(T(v)) == &T(v)

// 推荐场景
ptr := new(calculateValue())  // 计算值 - new 更简洁
ptr := &MyStruct{Field: 1}    // 字面量 - & 更常见
```

**性能**: 两者生成的汇编代码相同。

---

### Q4: 为什么 `new(nil)` 编译错误？

**A**: `nil` 不是值类型，没有具体类型：

```go
// 错误
ptr := new(nil)  // 编译错误: cannot use nil as type

// 正确
var p *int = nil
ptr := new(int(0))  // 创建指向 0 的指针
```

---

### Q5: new() 可以用于接口类型吗？

**A**: 需要类型断言：

```go
var i interface{} = 42

// 错误
// ptr := new(i)

// 正确
ptr := new(i.(int))
```

---

## 🧬 递归泛型

### Q6: 递归泛型和普通泛型有什么区别？

**A**:

| 特性 | 普通泛型 | 递归泛型 |
|------|----------|----------|
| 语法 | `T any` | `T C[T]` |
| 用途 | 通用容器/函数 | 自引用数据结构 |
| 复杂度 | 低 | 中高 |
| 典型场景 | Stack[T], Map[K,V] | Tree[T], Graph[N] |

---

### Q7: 为什么我的递归泛型编译错误？

**A**: 常见错误：

```go
// 错误1: 使用具体类型而非接口
type Bad[T Bad[T]] struct {}  // 编译错误

// 正确
type Good[T Good[T]] interface {}

// 错误2: 无限展开
type Infinite[T Infinite[T]] T  // 编译错误

// 正确
type Node[T Node[T]] interface {
    Children() []T  // 有限方法
}
```

---

### Q8: 递归泛型性能如何？

**A**:

- 编译期：类型检查稍复杂，但现代编译器优化良好
- 运行期：单态化生成，与普通泛型相同性能
- 相比 interface{}：无装箱开销，性能更好

## ⚡ 运行时

### Q9: Green Tea GC 需要手动启用吗？

**A**: **不需要**，Go 1.26 自动启用。

```bash
# 验证正在使用
$ go run -v your-program.go
# 输出中会显示 gc 相关信息
```

---

### Q10: 如何监控 Green Tea GC 性能？

**A**:

```go
import "runtime"

var m runtime.MemStats
runtime.ReadMemStats(&m)

fmt.Printf("GC次数: %d\n", m.NumGC)
fmt.Printf("GC暂停: %dµs\n", m.PauseNs[(m.NumGC+255)%256]/1000)
fmt.Printf("GC CPU: %.2f%%\n", m.GCCPUFraction*100)
```

**健康指标**: GC CPU < 10%，暂停 < 1ms

---

### Q11: 我可以禁用 Green Tea GC 吗？

**A**: **不可以**。但可以通过环境变量调整：

```bash
# 减少 GC 频率（更多内存使用）
export GOGC=200

# 增加 GC 频率（更少内存使用）
export GOGC=50

# 禁用 GC（仅调试，不推荐生产环境）
export GOGC=off
```

## 📦 标准库

### Q12: crypto/hpke 和 TLS 有什么区别？

**A**:

| 特性 | HPKE | TLS |
|------|------|-----|
| 层级 | 应用层加密 | 传输层安全 |
| 握手 | 1-RTT | 2-RTT (1.3为1-RTT) |
| 灵活性 | 高，可组合算法 | 固定套件 |
| 后量子 | 支持 ML-KEM | 逐步支持 |
| 使用场景 | 消息加密、文件加密 | 网络通信 |

---

### Q13: 何时使用 errors.AsType 而非 errors.As？

**A**:

```go
// 推荐：简洁，类型安全
if myErr, ok := errors.AsType[*MyError](err); ok {
    // myErr 已经是 *MyError 类型
}

// 旧方式：需要预先声明变量
var myErr *MyError
if errors.As(err, &myErr) { ... }
```

**建议**: 新项目使用 `AsType`，旧代码可保持 `As`。

---

### Q14: crypto 包随机参数变更影响我的代码吗？

**A**: 绝大多数情况下**不影响**。

**影响场景**:

- 依赖特定随机源进行确定性测试

**解决方案**:

```go
import "testing/cryptotest"

func TestDeterministic(t *testing.T) {
    cryptotest.SetGlobalRandom(yourTestRand)
    defer cryptotest.SetGlobalRandom(nil)

    // 测试代码
}
```

## 🛠️ 工具链

### Q15: go fix 会修改我的业务逻辑吗？

**A**: **不会**。go fix 只进行安全的语法转换：

- ✅ 循环 → `slices.Contains`
- ✅ if-else → `min`/`max`
- ✅ 旧 API → 新 API

所有修改保持语义等价。

---

### Q16: 如何编写自定义 Modernizer？

**A**:

```go
package mymodernizer

import "golang.org/x/tools/go/analysis"

var Analyzer = &analysis.Analyzer{
    Name: "mymodernizer",
    Doc:  "Description",
    Run:  run,
}

func run(pass *analysis.Pass) (interface{}, error) {
    // 实现分析逻辑
    // 使用 pass.Report 报告问题
    return nil, nil
}
```

然后使用 `go fix -fix=mymodernizer ./...` 运行。

---

### Q17: //go:fix inline 可以用于任何函数吗？

**A**: **不可以**。有严格限制：

```go
// ✅ 可以内联：简单转发
//go:fix inline
func Old() { New() }

// ❌ 不能内联：有副作用
//go:fix inline
func Bad() {
    println("side effect")
    DoSomething()
}

// ❌ 不能内联：复杂控制流
//go:fix inline
func AlsoBad() {
    for i := 0; i < 10; i++ { ... }
}
```

## 🔄 迁移问题

### Q18: 从 Go 1.25 迁移到 1.26 需要注意什么？

**A**:

1. **加密测试**: 检查是否依赖确定性随机
2. **JPEG 测试**: 更新图像相关测试的期望值
3. **GODEBUG**: 准备移除已弃用的设置
4. **运行测试**: `go test ./...`

**大多数情况下**: 直接升级，无需修改代码。

---

### Q19: 我的项目还在用 Go 1.20，如何规划升级？

**A**: 推荐升级路径：

```
Go 1.20 → 1.21 (PGO, min/max)
      → 1.22 (for循环改进)
      → 1.23 (迭代器)
      → 1.24 (性能优化)
      → 1.25 (容器感知)
      → 1.26 (new(expr), 递归泛型)
```

每步升级后运行测试，确保通过后再下一步。

## 🐛 调试问题

### Q21: 递归泛型类型推导失败怎么办？

**A**: 显式指定类型参数：

```go
// 类型推导失败
Traverse(root, callback)

// 显式指定
Traverse[*BinaryNode](root, callback)
```

---

### Q22: 如何调试 HPKE 加密失败？

**A**: 检查清单：

1. ✅ 发送方和接收方使用相同套件
2. ✅ 上下文信息（info参数）一致
3. ✅ 封装值（enc）正确传递
4. ✅ 私钥和公钥配对正确

```go
// 调试代码
fmt.Printf("Suite: %v\n", suite)
fmt.Printf("Enc size: %d\n", len(enc))
fmt.Printf("Ciphertext size: %d\n", len(ciphertext))
```

---

### Q23: go fix 修改了不想改的文件怎么办？

**A**:

```bash
# 1. 使用版本控制恢复
git checkout -- file.go

# 2. 使用 -n 预览，手动选择
go fix -n ./...
# 然后手动应用想要的修改

# 3. 使用 //go:build 跳过特定文件
//go:build !fix
package mypkg
```
