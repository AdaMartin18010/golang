# COMPARISON: Redis vs Memcached 缓存系统对比

> **维度**: Technology Stack
> **级别**: S (15+ KB)
> **标签**: #redis #memcached #cache #comparison #performance
> **权威来源**: [Redis Documentation](https://redis.io/docs/), [Memcached Wiki](https://github.com/memcached/memcached/wiki)

---

## 核心对比

| 特性 | Redis | Memcached |
|------|-------|-----------|
| **数据结构** | String, List, Hash, Set, ZSet, Stream, Bitmap, HyperLogLog | 仅 Key-Value |
| **持久化** | RDB, AOF | 不支持 |
| **集群** | Redis Cluster, Sentinel | 客户端分片 |
| **复制** | 主从复制 | 不支持 |
| **事务** | MULTI/EXEC | CAS (Check-and-Set) |
| **Lua 脚本** | 支持 | 不支持 |
| **Pub/Sub** | 支持 | 不支持 |
| **内存效率** | 中等 (数据结构开销) | 高 (简单 KV) |
| **单值大小** | 512 MB | 1 MB |
| **多线程** | Redis 6.0+ IO 多线程 | 完全多线程 |
| **适用场景** | 复杂数据结构、持久化、消息队列 | 纯缓存、简单 KV、高并发读取 |

## 性能对比

| 场景 | Redis | Memcached | 胜出 |
|------|-------|-----------|------|
| 简单 GET/SET | ~100k ops/s | ~200k ops/s | Memcached |
| 批量读取 | Pipeline 优化 | 多线程优势 | 相当 |
| 复杂操作 (ZADD/ZRANGE) | 原生支持 | 需应用层实现 | Redis |
| 内存效率 (小 value) | 对象开销 | Slab 分配 | Memcached |
| 内存效率 (大 value) | 高效 | 1MB 限制 | Redis |

## 选择建议

```
选择 Redis 如果你需要:
- 复杂数据结构 (List/Hash/Set/ZSet)
- 数据持久化
- 主从复制/高可用
- Lua 脚本
- Pub/Sub 消息
- 地理空间数据
- 流数据处理
- 事务支持

选择 Memcached 如果你需要:
- 纯缓存场景
- 简单 Key-Value
- 最大内存效率
- 高并发读取
- 多核 CPU 利用
- 易于水平扩展
- 与 MySQL 等集成
```

## 混合使用策略

```
多级缓存架构:

用户请求
    │
    ▼
┌─────────────┐     未命中    ┌─────────────┐
│  L1 Cache   │──────────────►│  L2 Cache   │
│ (Memcached) │               │   (Redis)   │
│ 热点数据     │               │ 全量缓存    │
└─────────────┘               └─────────────┘
    │                               │
    │ 命中                          │ 未命中
    ▼                               ▼
返回数据                      ┌─────────────┐
                              │   Database  │
                              └─────────────┘

策略:
- L1: Memcached (最热点，TTL 短)
- L2: Redis (全量，支持复杂查询)
- DB: 持久化存储
```
