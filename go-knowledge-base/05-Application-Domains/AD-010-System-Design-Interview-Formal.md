## 1. 形式化基础

### 1.1 系统设计面试定义

**定义 1.1 (系统设计面试)**
系统设计面试是评估候选人设计分布式系统能力的结构化对话，涵盖需求分析、架构设计、权衡分析和扩展性考虑。

**定义 1.2 (设计空间)**
设计空间 $D$ 是所有可能设计决策的集合：
$$D = \{d_1, d_2, ..., d_n\}$$

其中每个 $d_i$ 代表一个设计决策（如数据库选择、缓存策略等）。

**定理 1.1 (设计完备性)**
完备的系统设计必须涵盖：需求、架构、数据、扩展性、可靠性、运维。

### 1.2 RASCAL 框架

**定义 1.3 (RASCAL 框架)**

- **R**equirements (需求): 功能与非功能需求
- **A**rchitecture (架构): 高层组件设计
- **S**cale (规模): 容量规划与扩展策略
- **C**omponents (组件): 具体技术选型
- **A**lgorithms (算法): 核心算法设计
- **L**ogistics (运维): 监控、部署、故障处理

## 2. 需求分析形式化

### 2.1 功能需求

**定义 2.1 (功能需求)**
$$FR = \{f_1, f_2, ..., f_m\}$$

每个功能 $f_i$ 定义为：
$$f_i = \langle \text{输入}, \text{处理}, \text{输出} \rangle$$

### 2.2 非功能需求

**定义 2.2 (非功能需求)**
$$NFR = \{\text{可用性}, \text{一致性}, \text{延迟}, \text{吞吐量}, \text{可扩展性}\}$$

**可用性计算**:
$$A = \frac{MTBF}{MTBF + MTTR}$$

其中：

- MTBF: 平均故障间隔时间
- MTTR: 平均修复时间

| 可用性级别 | 年停机时间 | 应用场景 |
|------------|------------|----------|
| 99% (2个9) | 3.65 天 | 内部工具 |
| 99.9% (3个9) | 8.76 小时 | 电商平台 |
| 99.99% (4个9) | 52.6 分钟 | 支付系统 |
| 99.999% (5个9) | 5.26 分钟 | 关键基础设施 |

### 2.3 容量估算

**定义 2.3 (读写比例)**
$$R:W = \frac{\text{读请求数}}{\text{写请求数}}$$

常见系统比例：

| 系统类型 | 读:写 | 特点 |
|----------|-------|------|
| 社交媒体 | 100:1 | 读密集型 |
| 电商 | 10:1 | 中等读密集 |
| 日志系统 | 1:10 | 写密集型 |
| 聊天系统 | 1:1 | 平衡 |

## 3. 架构设计形式化

### 3.1 分层架构

**定义 3.1 (分层架构)**
系统分解为层次 $L_1, L_2, ..., L_k$，其中 $L_i$ 只依赖 $L_{i-1}$。

**经典 Web 架构层**:

```
┌─────────────────────────────────────┐
│  Layer 4: CDN / Edge                │
├─────────────────────────────────────┤
│  Layer 3: Load Balancer             │
├─────────────────────────────────────┤
│  Layer 2: Application Servers       │
├─────────────────────────────────────┤
│  Layer 1: Data Layer                │
│  (Database, Cache, Message Queue)   │
└─────────────────────────────────────┘
```

### 3.2 数据模型设计

**定义 3.2 (数据模型选择矩阵)**

| 数据特征 | 推荐存储 | 理由 |
|----------|----------|------|
| 结构化、事务性 | PostgreSQL | ACID、关系模型 |
| 高写入、时间序列 | Cassandra | 高吞吐、分区 |
| 文档型、灵活模式 | MongoDB | 灵活 Schema |
| 缓存、会话 | Redis | 内存、低延迟 |
| 搜索、全文检索 | Elasticsearch | 倒排索引 |
| 图关系 | Neo4j | 图遍历优化 |
| 对象存储 | S3/MinIO | 高可靠、低成本 |

### 4.3 常见系统设计对比

| 系统 | 核心挑战 | 关键技术 | 存储选型 |
|------|----------|----------|----------|
| **URL Shortener** | 短码生成、重定向 | Base62、哈希 | Redis + RDBMS |
| **Twitter Feed** | 扇出、Timeline | 推拉混合 | Redis + MySQL |
| **Chat/Messenger** | 实时性、在线状态 | WebSocket | Cassandra |
| **Rate Limiter** | 精确限流 | Token Bucket | Redis |
| **Key-Value Store** | 一致性、分区 | Raft | 自定义 |
| **Search Engine** | 索引、相关性 | 倒排索引 | Elasticsearch |
| **Video Streaming** | 带宽、自适应 | CDN、HLS | Object Storage |
| **Web Crawler** | 去重、礼貌性 | Bloom Filter | Kafka + DB |
| **Typeahead** | 低延迟前缀匹配 | Trie | Redis |
| **Notification** | 可靠投递、批处理 | 消息队列 | SQS/Kafka |

### 4.4 技术选型决策矩阵

```
需要选择数据库?
│
├── 需要事务支持?
│   ├── 是 → SQL 数据库
│   │       ├── 复杂查询? → PostgreSQL
│   │       ├── 读多写少? → MySQL + 读写分离
│   │       └── 地理分布? → CockroachDB/Spanner
│   │
│   └── 否 → NoSQL
│           ├── 键值查询? → Redis (内存) / DynamoDB (托管)
│           ├── 文档模型? → MongoDB
│           ├── 宽列? → Cassandra (写多) / HBase (扫描)
│           ├── 图查询? → Neo4j
│           └── 搜索? → Elasticsearch
│
需要缓存?
│
├── 数据大小?
│   ├── 可放入内存 → Redis
│   │       ├── 需要持久化? → Redis AOF/RDB
│   │       └── 需要集群? → Redis Cluster
│   │
│   └── 超内存 → CDN + 本地缓存
│           ├── 静态资源? → CloudFlare/Akamai
│           └── 动态内容? → 应用层缓存
│
需要消息队列?
│
├── 需求特征?
│   ├── 高吞吐、可丢消息 → Kafka
│   ├── 严格顺序 → Kafka (partition)
│   ├── 事务消息 → RocketMQ/Pulsar
│   ├── 简单任务队列 → RabbitMQ
│   └── 云原生托管 → SQS/SNS
│
需要负载均衡?
│
├── 部署环境?
│   ├── 云环境 → ELB/ALB (AWS), Cloud Load Balancing (GCP)
│   ├── Kubernetes → Ingress Controller
│   └── 自建 → Nginx/HAProxy
│       ├── L4 (TCP) → 性能优先
│       └── L7 (HTTP) → 内容路由
```

## 5. 典型系统设计详解

### 5.1 URL 短链服务

**需求**:

- 功能: 长 URL → 短 URL，重定向
- 规模: 100M 新 URL/天，读:写 = 100:1

**设计**:

```
┌──────────┐     ┌──────────────┐     ┌─────────────┐
│  Client  │────►│  Load        │────►│  Web        │
│          │     │  Balancer    │     │  Servers    │
└──────────┘     └──────────────┘     └──────┬──────┘
                                             │
                    ┌────────────────────────┼────────────────────────┐
                    ▼                        ▼                        ▼
            ┌──────────────┐       ┌────────────────┐      ┌──────────────┐
            │  Cache       │       │  Database      │      │  Key Gen     │
            │  (Redis)     │       │  (PostgreSQL)  │      │  Service     │
            └──────────────┘       └────────────────┘      └──────────────┘
```

**短码生成策略**:

| 策略 | 优点 | 缺点 |
|------|------|------|
| 哈希 (MD5) | 固定长度 | 碰撞风险 |
| Base62 自增 | 无碰撞 | 可预测 |
| 随机 + 检查 | 安全 | 生成较慢 |
| 预生成 | 快速 | 需要管理 |

### 5.2 News Feed (Twitter Timeline)

**推拉模式选择**:

| 模式 | 机制 | 适用场景 |
|------|------|----------|
| **推 (Push)** | 发布时推送到关注者 | 粉丝少的用户 |
| **拉 (Pull)** | 读取时拉取关注者内容 | 粉丝多的用户 |
| **混合** | 普通用户推，大V拉 | Twitter 实际方案 |

**Fan-out 计算**:

- 普通用户: 1000 粉丝 → 推模式可行
- 大V: 1M 粉丝 → 拉模式避免写入风暴

## 6. 面试技巧

### 6.1 时间分配

| 阶段 | 时间 | 活动 |
|------|------|------|
| 理解问题 | 2-3 min | 提问澄清 |
| 需求收集 | 5 min | 功能 + 非功能 |
| 高层设计 | 10-15 min | 画架构图 |
| 详细设计 | 15-20 min | 数据 + 算法 |
| 深度讨论 | 10 min | 面试官关注点 |
| 总结 | 2-3 min | 回顾关键点 |

### 6.2 常见陷阱与避免

| 陷阱 | 避免方法 |
|------|----------|
| 过早深入细节 | 先高层，再细化 |
| 忽视非功能需求 | 主动询问规模 |
| 单点故障 | 主动提出冗余 |
| 忽视扩展性 | 讨论增长策略 |
| 不提问 | 持续澄清需求 |
| 背模板 | 根据场景调整 |
